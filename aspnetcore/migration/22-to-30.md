## <a name="update-net-core-sdk-version-in-globaljson"></a>Обновление версии пакета SDK для .NET Core в файле global.json

```json
{
  "sdk": {
    "version": "3.0.100"
  }
}
```

### <a name="framework-reference"></a>Рамочная ссылка

Функции ASP.NET Core, которые были доступны через один из пакетов, перечисленных выше, доступны как часть `Microsoft.AspNetCore.App` общей платформы. *Общая платформа* — это набор сборок (*DLL*-файлы), которые установлены на компьютере, содержащий компонент среды выполнения и целевой пакет. Дополнительную информацию см. в этой публикации об [общей платформе](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/).

* Проекты, предназначенные для пакета SDK `Microsoft.NET.Sdk.Web`, неявно ссылаются на платформу `Microsoft.AspNetCore.App`.

  Для этих проектов не требуются дополнительные ссылки:

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Web">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>
      ...
  </Project>
  ```

* Проекты, `Microsoft.NET.Sdk` `Microsoft.NET.Sdk.Razor` которые нацелены или SDK, должны добавить явный: `FrameworkReference` `Microsoft.AspNetCore.App`

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Razor">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>

    <ItemGroup>
      <FrameworkReference Include="Microsoft.AspNetCore.App" />
    </ItemGroup>
      ...
  </Project>
  ```

#### <a name="framework-dependent-builds-using-docker"></a>Зависящие от рамок сборки с использованием Docker

Зависящие от рамок сборки консольных приложений, которые используют пакет, зависящий от [ASP.NET общей инфраструктуры](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/) Core может привести к следующей ошибке выполнения:

```console
It was not possible to find any compatible framework version
The specified framework 'Microsoft.AspNetCore.App', version '3.0.0' was not found.
  - No frameworks were found.
```

`Microsoft.AspNetCore.App`является общей структурой, содержащей ASP.NET основное время выполнения и присутствует только на [dotnet/core/aspnet](https://hub.docker.com/_/microsoft-dotnet-core-aspnet/) Docker изображение. 3.0 SDK уменьшает размер зависящих от фректов сборок, использующих ASP.NET Core, не включая дубликаты копий библиотек, доступных в общей структуре. Это потенциальная экономия до 18 МБ, но она требует, чтобы ASP.NET Core время выполнения присутствовать / установлен для запуска приложения.

Чтобы определить, есть ли у приложения зависимость (прямая или косвенная) от общей инфраструктуры ASP.NET Core, изучите файл *runtimeconfig.json,* созданный во время сборки/публикации вашего приложения. Следующий файл JSON показывает зависимость от общей инфраструктуры ASP.NET Core:

```json
{
  "runtimeOptions": {
    "tfm": "netcoreapp3.0",
    "framework": {
      "name": "Microsoft.AspNetCore.App",
      "version": "3.0.0"
    },
    "configProperties": {
      "System.GC.Server": true
    }
  }
}
```

Если приложение использует Docker, используйте базовое изображение, которое включает ASP.NET Core 3.0. Например, `docker pull mcr.microsoft.com/dotnet/core/aspnet:3.0`.

## <a name="kestrel"></a>Kestrel

### <a name="configuration"></a>Конфигурация

Мигрировать Kestrel конфигурации веб-хостинга, предоставляемых `ConfigureWebHostDefaults` *(Program.cs):*

```csharp
public static IHostBuilder CreateHostBuilder(string[] args) =>
    Host.CreateDefaultBuilder(args)
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.ConfigureKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseStartup<Startup>();
        });
```

Если приложение создает хост `HostBuilder`вручную с, `UseKestrel` вызов `ConfigureWebHostDefaults`на веб-хостинга строитель в:

```csharp
public static void Main(string[] args)
{
    var host = new HostBuilder()
        .UseContentRoot(Directory.GetCurrentDirectory())
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.UseKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseIISIntegration()
            .UseStartup<Startup>();
        })
        .Build();

    host.Run();
}
```

### <a name="connection-middleware-replaces-connection-adapters"></a>Подключение Middleware заменяет адаптеры подключения

Адаптеры`Microsoft.AspNetCore.Server.Kestrel.Core.Adapter.Internal.IConnectionAdapter`подключения () были удалены из Kestrel. Замените адаптеры подключения с помощью connection Middleware. Подключение Middleware аналогично HTTP Middleware в ASP.NET core конвейера, но для соединений более низкого уровня. HTTPS и журнал соединения:

* Были перемещены из адаптеров подключения в Connection Middleware.
* Эти методы расширения работают, как и в предыдущих версиях ASP.NET Core. 

Для получения дополнительной информации смотрите [пример TlsFilterConnectionHandler в разделе ListenOptions.Protocols статьи Kestrel](/aspnet/core/fundamentals/servers/kestrel?view=aspnetcore-3.0#listenoptionsprotocols).

### <a name="transport-abstractions-moved-and-made-public"></a>Транспортные абстракции перемещались и обнародовались

Транспортный уровень Kestrel предоставляется как открытый интерфейс в `Connections.Abstractions`. В рамках этих обновлений:

* `Microsoft.AspNetCore.Server.Kestrel.Transport.Abstractions`и связанные типы были удалены.
* <xref:Microsoft.AspNetCore.Server.Kestrel.KestrelServerOptions.NoDelay>был перемещен <xref:Microsoft.AspNetCore.Server.Kestrel.Core.ListenOptions> из вариантов транспортировки.
* `Microsoft.AspNetCore.Server.Kestrel.Transport.Abstractions.Internal.SchedulingMode`был удален <xref:Microsoft.AspNetCore.Server.Kestrel.KestrelServerOptions>из .

Для получения дополнительной информации смотрите следующие ресурсы GitHub:

* [Клиент/серверсетевые абстракции (dotnet/AspNetCore #10308)](https://github.com/dotnet/AspNetCore/issues/10308)
* [Реализация новой абстракции слушателя породы и повторноплайте Kestrel сверху (dotnet/AspNetCore #10321)](https://github.com/dotnet/AspNetCore/pull/10321)

### <a name="kestrel-request-trailer-headers"></a>Kestrel Запрос прицепзаголовков

Для приложений, ориентированных на более ранние версии ASP.NET Core:

* Kestrel добавляет HTTP/1.1 chunked заголовки прицепа в коллекцию заголовков запросов.
* Трейлеры доступны после того, как тело запроса читается до конца.

Это вызывает некоторые опасения по поводу двусмысленности между заголовками и прицепами, поэтому`RequestTrailerExtensions`трейлеры были перемещены в новую коллекцию () в 3.0.

HTTP/2 трейлеры запроса являются:

* Недоступно в ASP.NET Core 2.2.
* Доступно в 3.0 как `RequestTrailerExtensions`.

Для доступа к этим прицепам присутствуют новые методы расширения запросов. Как и в отношении HTTP/1.1, прицепы доступны после того, как тело запроса будет прочитано до конца.

Для выпуска 3.0 доступны `RequestTrailerExtensions` следующие методы:

* `GetDeclaredTrailers`&ndash; Получает заголовок запроса, `Trailer` который перечисляет, какие прицепы ожидать после тела.
* `SupportsTrailers`&ndash; Указывает, поддерживает ли запрос получение заголовков прицепов.
* `CheckTrailersAvailable`&ndash; Проверяет, поддерживает ли запрос трейлеры и доступны ли они для чтения. Эта проверка не предполагает, что Есть прицепы для чтения. Там может быть не прицепов `true` для чтения, даже если возвращается с помощью этого метода.
* `GetTrailer`&ndash; Получает запрошенный заголовок от ответа. Проверьте `SupportsTrailers` `GetTrailer`перед вызовом, <xref:System.NotSupportedException> или может произойти, если запрос не поддерживает заголовки задних.

Для получения дополнительной информации смотрите [трейлеры запроса Put в отдельной коллекции (dotnet/AspNetCore #10410)](https://github.com/dotnet/AspNetCore/pull/10410).

### <a name="allowsynchronousio-disabled"></a>AllowSynchronousIO отключен

Эти AAP являются источником голодания потоков, ведущих к сбоям приложений. В 3.0 `AllowSynchronousIO` отключен по умолчанию. ### <a name="security-middleware-guidance"></a>Руководство по промежуточного обеспечения безопасности

Поддержка авторизации и CORS унифицирована вокруг [промежуточного](xref:fundamentals/middleware/index) подхода. Это позволяет использовать те же промежуточное программное обеспечение и функциональность в этих сценариях. Обновленное промежуточное программное обеспечение авторизации предоставляется в этом выпуске, и CORS Middleware усиливается, чтобы он мог понять атрибуты, используемые контроллерами MVC.

#### <a name="cors"></a>CORS

Ранее CORS может быть трудно настроить. Среднее посуды было предусмотрено для использования в некоторых случаях использования, но MVC фильтры были предназначены для использования **без** промежуточного программного обеспечения в других случаях использования. С ASP.NET Core 3.0, мы рекомендуем всем приложениям, которые требуют CORS использовать CORS Middleware в тандеме с конечным пунктом routing. `UseCors`может быть предоставлена политика `[EnableCors]` по `[DisableCors]` умолчанию, а также атрибуты могут быть использованы для переопределения политики по умолчанию, когда это необходимо.

Рассмотрим следующий пример:

* CORS включен для всех конечных точек с указанной `default` политикой.
* Класс `MyController` отражает CORS `[DisableCors]` с атрибутом.

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseCors("default");

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute();
    });
}

[DisableCors]
public class MyController : ControllerBase
{
    ...
}
```

В более ранних версиях ASP.NET Core `[Authorize]` поддержка авторизации предоставлялась через атрибут. Разрешение промежуточного не было доступно. Программное обеспечение авторизации также может быть настроено с помощью политики по умолчанию, которая может быть отменена.

В ASP.NET Core 3.0 `UseAuthorization` или `Startup.Configure`позже, `HomeController` вызывается в , и следующее требует подписании в пользователе:

```csharp
    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints =>
    {
        endpoints
            .MapHealthChecks("/healthz")
            .RequireAuthorization(new AuthorizeAttribute(){ Roles = "admin", });
    });
}
```

Защита осуществляется для некоторых сценариев. Endpoints Middleware выбрасывает исключение, если авторизация или политика CORS пропущены из-за отсутствия промежуточного посуды. Ведется поддержка анализатора для предоставления дополнительной обратной связи о неправильной настройке.

Если приложение использует [пользовательские обработчики авторизации,](xref:security/authorization/policies#authorization-handlers)конечная точка, отправивая в обработчики другой тип ресурсов, чем MVC. Сначала поместите более конкретные маршруты (например, маршруты для области).

Рассмотрим следующий пример:

* `MapControllers`добавляет поддержку контроллеров, направив атрибуты.
* `MapAreaControllerRoute`добавляет обычный маршрут для контроллеров в области.


#### <a name="async-suffix-removal-from-controller-action-names"></a>Удаление суффикса Async из имен действий контроллера

В ASP.NET Core 3.0 ASP.NET Core `Async` MVC удаляет суффикс из имен действий контроллера. Это влияет на генерацию как реуктора, так и генерацию ссылок. Пример:

```csharp
public class ProductsController : Controller
{
    public async Task<IActionResult> ListAsync()
    {
        var model = await _dbContext.Products.ToListAsync();
        return View(model);
    }
}
```

До ASP.NET Ядра 3.0:

* Предыдущее действие можно получить на маршруте *Products/ListAsync.*
* Генерация ссылок `Async` требует указания суффикса. Пример:

    ```cshtml
    <a asp-controller="Products" asp-action="ListAsync">List</a>
    ```

В ASP.NET Core 3.0:

* Предыдущее действие можно получить на маршруте *«Продукты/Список».*
* Генерация ссылок не требует `Async` указания суффикса. Пример:

    ```cshtml
    <a asp-controller="Products" asp-action="List">List</a>
    ```

Это изменение не влияет на [`[ActionName]`](/dotnet/api/microsoft.aspnetcore.mvc.actionnameattribute) имена, указанные с помощью атрибута. Поведение по умолчанию может быть `Startup.ConfigureServices`отключено со следующим кодом в:

```csharp
services.AddMvc(options =>
    options.SuppressAsyncSuffixInActionNames = false);
```

#### <a name="changes-to-link-generation"></a>Изменения в генерации ссылок

Как поясняется в документации по [отличию от предыдущих версий реуктора,](xref:fundamentals/routing#differences-from-earlier-versions-of-routing)есть некоторые различия в генерации ссылок (использование `Url.Link` и аналогичные AIS, например). Сюда входит следующее.

* По умолчанию при использовании конечной точки параметры маршрута не обязательно сохраняются. Это поведение можно контролировать `IOutboundParameterTransformer` с помощью интерфейса.
* Создание URI для недействительного маршрута (контроллер/действие или страница, которая не существует) приведет к созданию пустой строки под маршрутизацией конечной точки вместо создания недействительного URI.
* Окружающие значения (параметры маршрута из текущего контекста) не используются автоматически в генерации ссылок с конечным маршрутизированием. Ранее при создании ссылки на другое действие (или страницу) из *текущих* значений окружающего маршрута будут выведены неопределенные значения маршрута. При использовании маршрутики конечных точек все параметры маршрута должны быть четко указаны во время генерации ссылок.

## <a name="opt-in-to-runtime-compilation"></a>Выберите компиляцию времени выполнения

До ASP.NET Core 3.0 компиляция представлений во времени выполнения была неявной особенностью платформы. Компиляция Runtime дополняет компиляцию представлений времен сборки. Это позволяет инфраструктуре компилировать представления и страницы Razor *(файлы .cshtml),* когда файлы изменены, без необходимости перестраивать все приложение. Эта функция поддерживает сценарий быстрого изменения iDE и обновления браузера для просмотра изменений.

В ASP.NET Core 3.0 компиляция времени выполнения является сценарием отказа. Компиляция времени сборки является единственным механизмом компиляции представления, включенным по умолчанию. Время выполнения зависит от Visual Studio или [дотнет-часы](xref:tutorials/dotnet-watch) в Visual Studio Code для восстановления проекта, когда он обнаруживает изменения в *файлах .cshtml.* В Visual Studio, изменения в *.cs*, *.cshtml*, или *.razor* файлы в проекте, который запускается (<kbd>Ctrl-F5</kbd>), но не отлажены (<kbd>F5</kbd>), вызвать перекомпиляцию проекта.

Для включения компиляции времени выполнения в проекте ASP.NET Core 3.0:

1. Установите пакет [Microsoft.AspNetCore.Mvc.Razor.RuntimeCompilation](https://www.nuget.org/packages/Microsoft.AspNetCore.Mvc.Razor.RuntimeCompilation) NuGet.
1. Обновление `Startup.ConfigureServices` для `AddRazorRuntimeCompilation`вызова:

    Для ASP.NET Core MVC используйте следующий код:

    ```csharp
    services.AddControllersWithViews()
        .AddRazorRuntimeCompilation(...);
    ```
 
    Для ASP.NET Core Razor Pages используйте следующий код:
 
    ```csharp
    services.AddRazorPages()
        .AddRazorRuntimeCompilation(...);
    ```
 
В выборке https://github.com/aspnet/samples/tree/master/samples/aspnetcore/mvc/runtimecompilation показан пример включения компиляции времени выполнения условно в средах разработки.

Для получения дополнительной информации о <xref:mvc/views/view-compilation>компиляции файлов Razor см.

## <a name="migrate-libraries-via-multi-targeting"></a>Мигрировать библиотеки через мультитаргетинг

Библиотекам часто требуется поддерживать несколько версий ASP.NET Core. Большинство библиотек, которые были составлены в отношении предыдущих версий ASP.NET Core, должны продолжать работать без проблем. Следующие условия требуют, чтобы приложение было перекрестным компилировано:

```xml
  </ItemGroup>

  <ItemGroup Condition="'$(TargetFramework)' == 'netstandard2.0'">

  </ItemGroup>

```

Используйте `#ifdefs` для включения ASP.NET основных AAP 3.0:

```csharp
var webRootFileProvider =

#else
#error unknown target framework
#endif
```
